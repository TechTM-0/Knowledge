# ベクトル検索の課題と改善提案

随時更新。実装したら「完了」マークを付けて経緯として残す。

## 根本課題：スコアの識別力が低い

### 実測データ（2026-05-03 時点、ノート 13 件）

- 全件のスコアが **0.48〜0.75（幅 0.27）に密集**
- 無関係なノートでも 0.5 前後のコサイン類似度になる（gemini-embedding-2 の高次元ベクトルの性質）
- 「最も関連するノート（0.75）」と「全く関係ないノート（0.48）」の差が 0.27 しかない

### なぜこれが問題か

コサイン類似度の絶対値に意味がないため、固定閾値（例: 0.6 以上を返す）が機能しない。
0.62 のノートが「関連あり」なのか「たまたま 0.62 になっただけの無関係なノート」なのか区別できない。

### 現在の対策：相対閾値

```python
# routers/vector_search.py
threshold = float(scores.max()) * 0.85
```

最高スコアを「このクエリでの天井」として、そこから 85% 未満を切り捨てる。
例：最高 0.80 → 0.68 未満を除外。クエリが変わっても基準が追従する。

**限界**：ノート数が少ないと最高スコアが低く、85% でも複数件残る（13 件全件返ってしまう事象が発生）。

## 改善提案

### ★ 優先度高：ハイブリッド検索（FTS5 + ベクトル RRF）

**なぜ有効か**  
ベクトルのスコアが 0.27 幅に密集する問題は、ベクトル単体では解決しにくい。
FTS5（キーワード完全一致）はまったく異なるスコア体系なので密集問題の影響を受けない。
Reciprocal Rank Fusion（RRF）で両方の「順位」を統合すれば、どちらでも上位のノートが強くブーストされる。

```python
# RRF スコア計算（k=60 が標準）
rrf_score = 1/(60 + rank_fts) + 1/(60 + rank_vector)
```

**実装箇所**：`routers/notes.py`（FTS5 検索のランク付き返却）、`routers/vector_search.py`（統合エンドポイント追加）、`app.js`（`searchFromApi` の切り替え）

---

### ★ 優先度中：スコア正規化（実装コスト低・即効性あり）

**なぜ有効か**  
0.48〜0.75 の狭いレンジを 0〜1 に引き伸ばすことで、閾値の設定がしやすくなる。
例：最低 0.48・最高 0.75 のとき、0.60 は `(0.60-0.48)/(0.75-0.48) = 0.44` と解釈できる。

```python
# routers/vector_search.py の scores 計算後に追加するだけ
min_s, max_s = scores.min(), scores.max()
if max_s > min_s:
    normalized = (scores - min_s) / (max_s - min_s)
```

**注意**：ノート数が少ないと min/max がノイズになる。ノートが増えるほど安定する。

---

### ★ 優先度中：タイトルと本文を別 embedding・重み付け

**なぜ有効か**  
現在はタイトル + 本文を結合して 1 ベクトルにしている。本文が長いと内容が薄まり全ノートで似通ったベクトルになりやすい。タイトルはノートの主題を端的に表すため、識別力が高い傾向がある。

```python
# 統合スコア例
score = 0.6 * title_similarity + 0.4 * body_similarity
```

**実装コスト**：`notes` テーブルに `title_embedding` カラム追加・再インデックスが必要。

---

### ★ 優先度低：クエリ拡張

**なぜ有効か**  
クエリが短い（1〜2 語）と embedding が曖昧なベクトルになる。Gemini で類義語・関連概念を生成してから embedding することで、より代表的なベクトルが得られる。

```python
expanded = gemini.generate(f"「{query}」の類義語・関連概念を5つ列挙")
query_vec = _embed(f"{query} {expanded}")
```

**実装コスト**：低（Gemini API 呼び出し 1 回追加）。ただし latency が増える。

---

### ★ 優先度低：閾値スライダー UI

スコアの可視化・ユーザーによる手動調整 UI。精度の根本解決にはならないがデバッグ・チューニング用途で有用。
バックエンドでスコア付き全件返却 → クライアント側フィルタリング（API 再呼び出し不要）の方式。
